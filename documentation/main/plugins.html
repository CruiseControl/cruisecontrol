CruiseControl Plugins

Introduction

It's tough to imagine all of the ways that CruiseControl can be used. Different tools and subtle nuances in team style make it difficult to code a "one-size-fits-all" solution. As a result, CruiseControl has been designed as a small core with a very high-level implementation of continuous integration, with most of the implementation details delegated to plugins. This makes it easy to create new functionality, modify existing functionality, or remove unwanted functionality. Plugins also create a shallower learning curve when digging in the codebase, as you can be assured that all of the cvs functionality, for example, exists in one place. Encapsulating all of the functionality into plugins also makes testing easier.

Plugin Design Overview

Plugins are represented by beans. Plugin classes are mapped to element names in the XML config file thanks to a process known as registration. Plugins distributed with cruisecontrol are all registered by default.

The configuration will be initiated by the ProjectXMLHelper class which handles the mapping of the registered plugin and delegates the plugin initialization to the PluginXMLHelper class. Once configured, the plugin is validated, and executed at the appropriate time in the build cycle. This execution time will depend on the plugin type.

Plugin Types

There are currently six different types of plugins that CruiseControl supports. Each has a slightly different role, but all are designed similarly. While they all implement different interfaces and have slightly different method names, we will refer to them collectively as plugins when discussing their similarities. The plugin types are:

  • Bootstrapper: run before the build
  • SourceControl: poll a code repository and determine if anything has changed
  • Builder: perform the actual building and testing of your code
  • LabelIncrementer: handles incrementing the label for you to use to tag your source
  • Publisher: publish the results of your build, via email for example
  • Listener: handles project events

Plugin Initialization

The PluginXMLHelper will use introspection to configure the bean.

Plugin configuration

In the case of normal initialization, the PluginXMLHelper will call setter methods that correspond to attributes on your xml tag in the config file. For example, the method setTarget(String target) on the AntBuilder class corresponds to the attribute target on the <ant/> element in the config file. The parser is case insensitive, so the case need not match from your attributes to your method names, so the standard capitalization rules should apply.

It is possible for you to declare nested elements within your plugin's declaration in the config file. In this case, CruiseControl will expect your plugin class to implement an add or create method for each nested element, depending whether your nested element is itself a plugin or not:

  • for nested elements that are also plugins, the PluginXMLHelper doesn't know to which class the element should map and then asks the ProjectXMLHelper to configure the element.
  • for simple (non plugins) nested elements, the plugin knows the type that will map to the config element name. Returning to our example AntBuilder class, we will see a createJVMArg() method. We'll also see that AntBuilder has a nested JVMArg class. This create method is responsible for creating an instance of JVMArg and keeping a reference to that object for AntBuilder to use later. It returns a reference to the newly created JVMArg object, so that PluginXMLHelper can configure the nested element in the same way as the parent element. In theory, there is no limit to the depth of your nesting, as long as the appropriate create methods have been written.

Plugin Validation

Immediately after the parser has configured your plugin, we will validate the plugin using it's own validate() method. This will enable us to fail quickly if your plugin has been incorrectly configured, saving you the time of waiting for your plugin to execute to determine whether your CruiseControl installation has been successful.

Plugin Execution

After we have configured and validated all of the plugins, we will begin executing them according to their type. Each plugin has an "action" method that is the one method that CruiseControl will call when executing your plugin. Let's take the example of the bootstrappers. At the beginning of the build cycle, the first thing that we need to do is to run the bootstrappers. We've already initialized and validated each of these, so now we will just call the bootstrap() method as we iterate over the list of registered bootstrappers. It's fairly similar for each of the other types of plugins.

Plugin Registration

The registration of plugins is a simple and straightforward task. Directly under the <cruisecontrol> element, or within a <project> element, simply add a <plugin name="" classname=""/> element, where the name corresponds to the name you wish to use within the config file, and the classname corresponds to the actual plugin Java class.

Plugins specified directly under the root element will be available to all your projects, plugins under a project element will be only available within that project. This is useful to override a plugin like the labelincrementer for a single project.

In the interest of keeping config file sizes to a minimum, all plugins that ship with CruiseControl will be automatically registered. Should you wish to use one of the registered plugin names with your own custom plugin class, you can just explicitly register the plugin name and that will override the default registration.

Plugin Preconfiguration

On registering plugins, you can also define default values for some of the properties and nested elements of the plugins. You can do this by simply using the same attributes and child nodes as you would when using the plugin itself.

Defaults can be overridden when you actually use the plugin within a project. Note: when nested elements have a multiple cardinality, overriding an already pre-configured child element might accumulate them.

If you want to preconfigure an already registered plugin (whether default plugin, or plugin registered at the root of the config), you may leave out the classname attribute of the plugin element.

An example: to define some default values for the htmlemail publisher, you can specify something like this in your config file:

  <plugin name="htmlemail"
          mailhost="smtp.example.com"
          returnaddress="buildmaster@example.com"
          subjectprefix="[CC]"
          xsldir="C:\java\cruisecontrol-2.2.1\report\jsp\webcontent\xsl"
          css="C:\java\cruisecontrol-2.2.1\report\jsp\webcontent\css\cruisecontrol.css">
     <always address="project-dev@domain.com"/>
  <plugin/>

Combined with ant style properties this can greatly reduce the size of your config files. In the extreme case you can create a template project where all that needs to be supplied is the project name:

<cruisecontrol>
  <plugin name="project">
    <bootstrappers>
      <cvsbootstrapper localworkingcopy="projects/${project.name}"/>
    </bootstrappers>
    ...
  </plugin>

  <project name="foo" />
  <project name="bar" />
  ...
</cruisecontrol>

More information and examples can be found on the wiki.

Plugin Inheritance

Plugin pre-configuration is very powerful scheme for the significant simplification of project configuration, especially when more projects share same part of configuration

The only disadvantage of the "raw" preconfiguration is, that although several plugins differing by their name but based on the same class (either using classname="..." or from="..." plugin attributes) can be defined, all the plugin "instances" are mutually independent. It means that if a plugin with only "slightly different" configuration needs to be created, which leads to the repetition being harder to maintain. And it is the case where plugins "inheritance" can be very useful.

For example, have an email with quiet complex address mapping and other configuration, but without a <always/>, <ignore/> and other attributes, since they are project-specific:

<plugin name="email" returnaddress="..." mailhost="..." subjectprefix="...">
  <map     alias="..." address="..."/>
  <map     alias="..." address="..."/>
  ...
  <propertiesmapper file=""/>
  ...
</plugin>

<project name="proj1">
  ...
  <publishers>
    <email> <always address="maintainer1"/><always address="leader1"/> ...
            <ignore user="..."/><ignore user="..."/><ignore user="..."/>...
    </email>
  </publishers>
</project>
<project name="proj2">
  ...
  <publishers>
    <email> <always address="maintainer2"/><always address="leader2"/> ...
            <ignore user="..."/><ignore user="..."/><ignore user="..."/>...
    </email>
  </publishers>
</project>
...

This will work pretty well, the problem starts with many projects where several share equal <email/> setting (e.g. a subset of projects the same people are working on) - the project-specific settings will be copied again and again. However, with the inheritance we can define a per-group configuration:

<plugin name="email" returnaddress="..." mailhost="..." subjectprefix="...">
  <!-- the same as in the previous example -->
</plugin>

<plugin name="email_g1" from="email" inherits="email">
  <always address="maintainer1"/><always address="leader1"/> ...
  <ignore user="..."/><ignore user="..."/><ignore user="..."/>...
</plugin>
<plugin name="email_g2" from="email" inherits="email">
  <always address="maintainer2"/><always address="leader2"/> ...
  <ignore user="..."/><ignore user="..."/><ignore user="..."/>...
</plugin>

<project name="proj1">
  ...
  <publishers> <email_g1/>
  </publishers>
</project>
<project name="proj1.1">
  ...
  <publishers> <email_g1/>
  </publishers>
</project>
<project name="proj2">
  ...
  <publishers> <email_g2/>
  </publishers>
</project>
...

It is clear that without inherits="" attribute, the base email setting and addresses mapping would not be defined in any of the email_g* plugins.

The inheritance is carried out entirely on XML level (in PluginRegistry), while the following rules are applied:

  • all parent's attributes not defined in the child are copied to the child,
  • all nested parent's XML elements are copied to the beginning of the child node, i.e. before the first nested child's element.

Now to make something bit more tricky. What if we need one single project to run for various configurations? It would be really nice to have the project defined only once, and only feed it by various configurations. Well, with the use of inheritance we can do the following: 

<plugin name="build1" from="ant"  target="...">
  <property name="foo1" value="bar1.1">
  <property name="foo2" value="bar1.2">
  <property name="foo3" value="bar1.3">
  <env      name="env1" value="val1.1">
</plugin>
<plugin name="build2" from="ant"  target="...">
  <property name="foo1" value="bar2.1">
  <env      name="env1" value="val2.1">
</plugin>
<plugin name="build3" from="ant" inherits="conf2" target="...">
  <jvmarg   arg="...">
  <property name="foo2" value="bar3.1">
  <property name="foo3" value="bar3.3">
  <env      name="env1" value="val3.1">
</plugin>

<plugin name="project">
  ...

  <schedule>
    <ant/>
  </schedule>
</plugin>

<project name="p1">
  <plugin name="ant" inherits="build1"/>
</project>
<project name="p2">
  <plugin name="ant" inherits="build2"/>
</project>
<project name="p3">
  <plugin name="ant" inherits="build3"/>
</project>

It pre-configures three ant builders named build1, build2 and build2, where the first two are independent on each other, but the third contains all the definitions from the second, plus adds some new stuff. Finally, we define a project using a "generic" <ant/> builder.
However, the line <plugin name="ant" inherits="..."/> defines project-local plugin of <ant/> builder, inherited from the required pre-configured ant builder. This local <ant/> is then used in the schedule, as required by the project definition.

BE CAREFUL:
it is not precisely defined what will happen when overriding an already defined child element (since the inheritance is carried out on XML level) - The resulting behavior is entirely module-dependent and may sometimes lead to some kind of "already defined" error (i.e. it is up to the particular plugin implementation to handle the repeated options correctly). It is highly recommended to study and test the particular modules before using complex inheritance schemes.

Plugin XML Documentation

The config reference is automatically generated from Java annotations contained in the plugin code. Developers of CruiseControl plugins should include all the applicable documentation annotations in their code to ensure that their plugins will be properly documented and will work correctly with any third-party tools that use these annotations.

When CruiseControl parses a plugin class to generate documentation and other metadata, it looks at the class's methods to determine the plugin's allowable attributes and children. Methods beginning with "set" are interpreted as attributes, and methods beginning with "add" or "create" are interpreted as children. Annotations describing the attributes and children are placed on these methods.

Here's a simple self-documenting example: 
import net.sourceforge.cruisecontrol.gendoc.annotations.*;

@Description("This will make it into the plugin description. " +
    "<b>You can also have HTML code.</b>")
public class MyPlugin {

    /**
     * A method starting with "set" will become an entry in the attribute table for
     * the plugin.
     */
    @Description("Detailed description of this attribute")
    @Title("A concise, human-readable title")
    @DefaultValue("defaultValue")
    @Required
    public void setMyField(String myField) { ... }

    /**
     * A @SkipDoc annotation instructs gendoc to ignore a method it would otherwise
     * try to interpret.
     */
    @SkipDoc
    public void setThread(Thread t) { ... }

    /**
     * A method starting with "set" or "create" will become an entry in the child
     * table for this plugin.
     */
    @Description("description")
    @Title("title")
    @Cardinality(0, -1) // Indicates any nonnegative number of children may occur.
    public void addSourceControl(SourceControl s) { ... }

}

The following annotations are permitted on different code elements. No annotations are required, as the parser can infer the presence and types of attributes and children directly from the Java source. However, it is recommended that each plugin class and attribute/child method be annotated at least with @Description or @DescriptionFile.

  • Plugin Class
    • @Description
    • @DescriptionFile
  • Attribute set Method
    • @Description
    • @DescriptionFile
    • @Title
    • @Cardinality
    • @Required
    • @Optional
    • @Default
  • Child add or create Method
    • @Description
    • @DescriptionFile
    • @Title
    • @Cardinality
  • Child create Method
    • @ManualChildName

To save you from having to write long strings of HTML inside Java annotations, the @DescriptionFile annotation is also provided, which loads documentation from an HTML file bundled in the same JAR as the Java class files. Details can be found in the javadoc for this annotation, but here are documented some accepted "best practices" regarding its use:

  • Don't use @DescriptionFile if you can put all the documentation in the Java source without inconvenience. Too many extra HTML files clutter the code folders.
  • Whenever possible, use @DescriptionFile without any arguments. To make sure the PluginInfoParser finds the correct file when you do this, place your HTML documentation file as noted below:
    • If annotating a plugin class, place the HTML file in the same source folder as the class and name it <CLASS>.html, where <CLASS> is the name of the class (without package information).
    • If annotating a plugin method (beginning with add, set, or create), place the HTML file in the same source folder as the class and name it <CLASS>.<METHOD>.html, where <CLASS> is the name of the class (without package information) and <METHOD> is the name of the annotated method.

For more information, see the JavaDoc for these annotations, in the net.sourceforge.cruisecontrol.gendoc.annotations package. The class responsible for reading these annotations is the PluginInfoParser class.

NOTE. Since only a few plugins have so far been documented using the annotations described above, the project build scripts are configured to write the generated HTML doc to "/documentation/main/configxml-gendoc.html". The website still links to the old, manually edited doc (configxml.html). You are encouraged to help with the transition by moving content from this old doc into source code annotations. You may use DefaultPropertiesPlugin and CruiseControlConfig and as models for this.
